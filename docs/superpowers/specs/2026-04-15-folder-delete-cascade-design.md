# 文件夹级联删除设计

## 目标

将“删除文件夹”的行为从“删除文件夹并把项目移到未分类”改为“删除文件夹时，同时永久删除该文件夹下的全部项目和素材”。

本次改动只调整文件夹删除语义，不改变单个项目删除语义，也不引入回收站、软删除或恢复能力。

---

## 范围

**纳入范围：**
- 删除文件夹时，级联删除该文件夹下的全部项目
- 级联删除这些项目的数据库记录与项目级文件资源
- 删除文件夹记录本身
- 更新文件夹列表中的确认文案，明确提示“永久删除全部项目和素材，无法恢复”

**不纳入范围：**
- 回收站
- 软删除
- 恢复操作
- 选择性保留文件夹内项目
- 删除前输入文件夹名二次确认
- 未分类迁移逻辑保留为可选行为

---

## 架构

采用“文件夹删除复用项目删除”的方案，而不是为文件夹级联删除单独复制一套项目清理逻辑。

现有基础能力：
- `deleteProject(id)` 已能删除项目记录
- `deleteProject(id)` 已能清理项目级图片与导出目录
- 文件夹删除 API 已单独存在

实现方式是把 `deleteFolder(folderId)` 改为异步流程：

1. 查询该文件夹下的全部项目
2. 逐个调用现有 `deleteProject(projectId)`
3. 所有项目删除完成后，再删除文件夹记录

这样可以保证项目级联删除的数据库与文件清理逻辑只有一份，降低遗漏和行为漂移风险。

---

## 当前问题

当前 `deleteFolder(folderId)` 的行为是：

```text
projects.folder_id = null
→ 删除 folder 记录
```

这会导致：
- 用户删除文件夹后，项目仍然保留在“未分类”
- 行为与“删除文件夹”直觉不一致
- UI 虽然提示了“移到未分类”，但和用户真实预期冲突

---

## 目标行为

删除文件夹时，系统应执行以下语义：

```text
删除文件夹
  → 删除该文件夹下全部项目
    → 删除项目下游数据库数据
    → 删除项目图片目录
    → 删除项目导出目录
  → 删除文件夹记录
```

删除完成后：
- 文件夹不再存在
- 文件夹下项目不再存在
- 不会出现在“未分类”
- 项目文件资源不会残留

---

## 数据流

### 后端删除链路

```text
DELETE /api/folders/[id]
  → deleteFolder(folderId)
    → 查询 projects where folder_id = folderId
    → for each project: await deleteProject(project.id)
    → 删除 project_folders 记录
  → 返回 { success: true }
```

### 失败处理

如果删除某个项目时失败：
- 停止后续删除
- 保留文件夹记录
- API 返回错误
- 前端提示“删除文件夹失败”或后端返回的具体错误

不允许出现“部分项目已删、文件夹也被删掉”的中间状态。

---

## API 与数据层变更

### `deleteFolder(folderId)` 改为异步

当前为同步函数，需要改为 `async`，因为内部将调用 `await deleteProject(...)`。

职责：
- 加载该文件夹下项目列表
- 顺序执行项目删除
- 在全部成功后删除文件夹

顺序删除优先于并发删除，理由是：
- 现有项目删除涉及数据库与文件系统
- 顺序执行更容易保证失败时的状态可理解
- 当前项目量预期不大，优先选择行为确定性

### `DELETE /api/folders/[id]`

当前 API 需要从：

```ts
deleteFolder(id);
```

改为：

```ts
await deleteFolder(id);
```

确保文件清理完成后再返回成功。

---

## UI 变更

### 文件夹列表确认文案

当前文案：

```text
确定删除文件夹「X」吗？文件夹内的项目会移到"未分类"。
```

改为更强提醒：

```text
确定删除文件夹「X」吗？会同时永久删除该文件夹下的全部项目和素材，无法恢复。
```

目标是避免用户误以为这只是“解散分组”操作。

### 项目删除文案

保持不变。单个项目删除仍然是项目级彻底删除。

### 手动验证

1. 创建一个带 1 个项目的文件夹并删除
   - 文件夹消失
   - 项目不出现在未分类

2. 创建一个带多个项目的文件夹并删除
   - 文件夹和项目全部消失

   - 文件夹被正常删除
   - 不报错

## 风险与控制

### 风险 1：重复实现项目删除逻辑导致遗漏

控制：
- 文件夹删除直接复用 `deleteProject()`
- 不在 `deleteFolder()` 中手写项目文件删除逻辑

### 风险 2：文件夹先删，项目残留

控制：
- 删除顺序固定为“项目 -> 文件夹”
- 只有所有项目删除成功后才删文件夹

### 风险 3：用户误删

控制：
- 强化确认文案
- 明确写出“永久删除”和“无法恢复”

## 实施边界

本次目标是把文件夹删除语义调整为真正的级联删除，不扩展成完整的资源回收管理系统。

因此：
- 不做软删除
- 不做恢复
- 不做批量预览将被删除的项目列表
- 不做更复杂的二次确认交互

后续如果用户需要“误删可恢复”，再单独设计回收站方案。
