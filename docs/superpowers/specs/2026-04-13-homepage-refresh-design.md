# 首页改版设计稿

## 背景

当前首页已经具备暖色品牌风格和基础信息结构，但存在三个问题：

- Hero 区文案偏开发视角，用户进入页面后无法快速理解页面用途。
- 新建项目交互放在页内输入框中，打断了首页的视觉重点。
- 概览卡和项目列表包含较多解释性信息，影响页面的清晰度和操作聚焦。

本次改版目标是不改变现有暖色品牌气质，只通过结构收敛和文案重写，让首页更像一个运营可直接使用的工作台。

## 目标

- 保留当前暖灰米色背景、橙色品牌色、圆角卡片和柔和阴影。
- 将首页收敛为三个层级：头部操作区、数据概览区、项目列表区。
- 将“新建项目”从页内输入改为按钮触发的输入弹窗。
- 删除开发语义和技术提示，改成人话文案。
- 强化项目名称与主要动作，弱化次要信息和危险操作。

## 非目标

- 不改动项目工作区页面。
- 不改动项目数据结构、API 接口和创建后跳转逻辑。
- 不引入新的图标库、组件库或复杂动效系统。
- 不切换整体视觉风格到冷色后台风。

## 设计方案

### 1. 页面结构

首页保持三段式结构：

1. Hero 区
2. Dashboard 概览区
3. 项目列表区

页面整体仍使用当前的大圆角、白卡片、细边框和轻阴影体系，但删除与页面主任务无关的辅助说明。

### 2. Hero 区

Hero 区保留大卡片形式，左侧为页面定位，右侧为单一主操作。

文案定稿如下：

- 标签：`图文提效工作台`
- 标题：`AI 图文生产工作台`
- 副标题：`集中管理您的 AI 图文创作项目与素材资产。`
- 主按钮：`＋ 新建项目`

视觉要求：

- 保留暖色光晕和轻装饰，但降低存在感，避免分散注意力。
- 去掉当前偏“阶段说明”式的长文案。
- 主标题不再拆成长句强调，改为直接表达工具用途。
- 右侧不再展示常驻输入框，只保留一个高权重主按钮。

### 3. 新建项目弹窗

点击 `＋ 新建项目` 后打开居中模态弹窗。

弹窗内容：

- 标题：`新建项目`
- 说明：`输入项目名称后即可进入工作台继续编辑。`
- 输入框占位：`例如：Q2-期中冲刺拍题精学`
- 按钮：`取消`、`创建项目`

交互要求：

- 输入为空时阻止提交，并显示明确错误文案。
- 提交成功后沿用现有行为：创建项目、跳转到新项目工作区、刷新路由。
- 提交失败时展示接口错误或兜底错误文案。
- 支持遮罩层关闭和显式取消关闭。
- 打开弹窗后输入框应可立即输入，关闭后状态重置到合理范围。

实现方式：

- 现有 `CreateProjectForm` 从页内表单改为“按钮 + 弹窗表单”。
- 如项目内没有合适的基础弹窗组件，可新增一个轻量通用弹窗组件到 `components/ui/`。
- 不引入外部依赖。

### 4. Dashboard 概览区

保留 4 张并排概览卡，突出数字和名称，不展示额外解释性小字。

卡片文案：

- `项目总数`
- `活跃项目`
- `方向卡总数`
- `文案卡总数`

视觉要求：

- 卡片继续使用纯白背景和弱阴影。
- 数字是主视觉，字号明显大于说明文字。
- 说明文字保持浅灰且简洁。
- 可增加极轻量的装饰圆点或色块，不引入复杂图形系统。

数据来源保持现状：

- 项目总数：项目数量
- 活跃项目：状态为 `active` 的项目数量
- 方向卡总数：所有项目方向卡数量总和
- 文案卡总数：所有项目文案卡数量总和

### 5. 项目列表区

区块标题仅保留：

- 标题：`项目列表`

删除当前右上角技术提示文案，例如本地 SQLite 提示。

单行列表结构：

- 左侧第一行：项目名 + 状态标签
- 左侧第二行：`图文 · 方向 X 条 · 文案 X 条`
- 右侧操作区：更新时间、删除、进入

状态标签：

- `active` 显示为 `进行中`
- `draft` 显示为 `草稿`

动作文案：

- `删除`
- `进入 >`

视觉要求：

- 项目名加粗，作为列表主层级。
- 状态标签缩小，避免抢占标题注意力。
- 行 hover 保留轻微抬升和边框强调，但不做明显整行染色。
- `删除` 默认低对比，仅在 hover 当前行时变清晰，降低列表噪音。
- `进入 >` 改为明确可读的次级动作，不再依赖仅图形化的圆形箭头按钮。
- 时间信息维持最弱层级，放在右侧。

空状态文案：

- 标题：`还没有项目`
- 说明：`新建一个项目，开始整理图文方向与文案内容。`

### 6. 可访问性与交互细节

- 弹窗需包含 `role="dialog"` 和 `aria-modal="true"`。
- 弹窗打开后，应把焦点导向输入框或弹窗容器内首个可交互元素。
- 键盘用户需可通过 `Esc` 关闭弹窗。
- 所有按钮保持现有 focus-visible 风格。
- 删除和进入操作不应因为 hover 优化而失去键盘可达性。

### 7. 实现边界

本次实现只应修改首页相关代码，预计涉及：

- `app/page.tsx`
- `components/dashboard/create-project-form.tsx`
- `components/dashboard/project-list.tsx`
- 如有必要，新增 `components/ui` 下的轻量弹窗组件

不应扩散到工作区、数据层、数据库 schema 或项目 API 逻辑。

### 8. 验证方案

实现完成后执行：

- `npm run lint`
- `npm run typecheck`

如果首页相关 source test 或断言测试因为文案、结构或交互变化失败，应同步更新测试，前提是测试覆盖的行为仍符合本设计。

## 风险与取舍

- 将页内输入改为弹窗后，首页表面信息会更干净，但多了一次点击。这是有意取舍，用来换取更明确的页面焦点。
- 删除技术说明会降低对开发状态的展示，但更符合首页面向运营用户的目标。
- 不引入额外图标库和重型弹窗依赖，可降低实现成本，但视觉表达会保持克制而非彻底重做。

## 实施建议

优先顺序如下：

1. 重写 Hero 文案并替换新建交互为弹窗
2. 精简概览卡信息层级
3. 重排项目列表动作与 hover 状态
4. 跑 lint、typecheck 和相关测试

该设计默认沿用现有暖色品牌设计系统，只做信息架构和交互层级优化。
