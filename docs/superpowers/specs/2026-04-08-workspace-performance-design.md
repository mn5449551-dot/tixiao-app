# 工作区性能重构 Design

**Date:** 2026-04-08

**Goal:** 在不改变现有功能和业务结果的前提下，提升项目页首屏加载速度、工作区交互流畅度，以及生成过程中的稳定性。

## 背景与现状

当前项目页的性能瓶颈集中在 3 个方面：

1. `app/projects/[id]/page.tsx` 通过 `getProjectWorkspace()` 一次性查询并下发完整 `workspace`，包含项目、需求卡、方向、文案卡、图片配置、图片组和图片列表。
2. `WorkspaceShell`、`WorkflowCanvas`、`ProjectTree`、`AgentPanel` 共同依赖这份大对象，任意局部更新都容易触发整页重算与多区域重渲染。
3. 多个交互路径通过 `router.refresh()` 或 `canvas-refresh` 驱动刷新，生成轮询也每 3 秒整页刷新一次，导致小更新放大为整页刷新。

构建基线：

- `next build` 可通过，但存在 1 个 Turbopack/NFT tracing warning，来源是导出链路的文件系统追踪范围过大。
- 运行时性能问题比构建性能问题更突出，优先级应放在页面拆分、数据粒度和刷新策略。

## 目标与非目标

### 目标

- 降低项目页首屏等待时间，让页面框架和关键内容更快可交互。
- 降低画布、左树、右侧助手之间的相互拖累，让局部操作只更新局部。
- 将图片生成轮询从整页刷新改为轻量状态更新，减少抖动和卡顿。
- 保持现有 API 语义、页面功能与用户操作路径不变。

### 非目标

- 不修改业务流程、卡片字段、AI 生成逻辑或画布信息结构。
- 不引入新的后端服务或远程缓存。
- 不做全仓库级的大规模 UI 改版。

## 可选方案

### 方案 A：轻量优化

做组件懒加载、局部 `memo`、少量 effect 清理和 tracing warning 修复。

优点：

- 风险最低，改动范围小。

缺点：

- 无法解决整页刷新和大对象下发的问题，收益有限。

### 方案 B：结构优化，功能不变

将工作区拆为稳定壳层和独立数据区块；把大查询拆成多个明确查询；把整页刷新替换为局部数据失效与局部更新。

优点：

- 能直接命中当前主要瓶颈，收益与风险比较平衡。

缺点：

- 需要调整页面数据边界与部分客户端状态组织。

### 方案 C：深度状态平台化重构

在方案 B 基础上建立更完整的规范化 store、统一事件层和更细的数据同步模型。

优点：

- 长期上限最高。

缺点：

- 周期长、回归面更大，不适合当前以“尽快提升体感”为主的目标。

**推荐：方案 B。**

## 推荐架构

### 页面分层

将当前项目页改为以下结构：

1. `ProjectWorkspacePage`
   仅获取项目头部所需的轻量数据，例如 `projectId`、标题、状态。
2. `WorkspaceShell`
   只负责三栏布局、折叠状态、加载边界与懒加载，不直接持有整份业务数据。
3. `ProjectTreePanel`
   只请求树结构所需数据。
4. `WorkflowCanvasPanel`
   只请求画布节点、边、节点状态与最小交互元数据。
5. `AgentPanel`
   保持助手独立请求模型，仅与自身会话及必要的工作区失效信号耦合。

### 数据访问层

保留现有 `project-data.ts` 作为业务实现来源，但新增更细粒度的读取接口，避免 `getProjectWorkspace()` 成为运行时主通道：

- `getWorkspaceHeader(projectId)`
- `getProjectTreeData(projectId)`
- `getCanvasData(projectId)`
- `getGenerationStatusData(projectId)`
- `getAssistantData(projectId)` 或沿用当前助手专用 API

`getProjectWorkspace()` 可保留作为兼容入口，但不再用于项目页主渲染链路。

## 刷新与状态模型

### 局部失效替代整页刷新

将当前的整页刷新模式改成按区域失效：

- 左树变更只刷新左树数据。
- 画布节点相关变更只刷新画布数据。
- 助手确认填充后，触发需求卡、左树、画布的局部刷新，而不是整个项目页刷新。

### 轮询模型调整

当前 `useGenerationPolling()` 会在存在生成中图片时每 3 秒执行 `router.refresh()`。改造后应改为：

- 轮询轻量状态接口，只拿图片状态、错误信息和新文件 URL。
- 将状态合并进画布局部数据源。
- 仅在必要时刷新左树计数或节点摘要，不刷新助手面板。

### 客户端共享状态边界

不引入复杂的全局状态平台，仅维护最小共享状态：

- 当前高亮节点 ID
- 画布数据版本或失效标记
- 树数据版本或失效标记
- 当前是否有进行中的图片轮询

共享状态应以 UI 协调为主，不承载完整业务数据。

## 组件与渲染优化

### 懒加载

对以下重量组件增加懒加载边界：

- `WorkflowCanvas`
- `AgentPanel`
- `InpaintModal`
- 图片预览相关弹层

首屏先显示稳定壳层与必要骨架，再按需加载重量区块。

### 画布渲染收敛

画布优化重点：

- 避免因整份 `workspace` 对象变化而重建整张图。
- 将节点构建数据源限制在画布专属查询结果。
- 保留当前高亮、布局和已展开层级等 UI 状态，不因局部数据刷新而全部回退。
- 减少在 render 路径里对 `canvasNodes.find()` 这类线性查找的重复执行，必要时改为基于 map 的索引。

### 左树与助手去耦

- 左树不再依赖完整画布数据，只消费树所需字段与统计信息。
- 助手不再依赖完整工作区对象，仅依赖 `projectId` 与会话接口；确认成功后通过局部失效通知其他区域更新。

## 构建与服务端优化

### 查询拆分

`getProjectWorkspace()` 当前存在明显的逐层嵌套查询模式。拆分后：

- 页面首屏只执行轻量头部查询。
- 左树查询只取树结构必要字段。
- 画布查询只取画布节点和图片状态需要的字段。
- 生成轮询查询只取状态字段。

这会同时降低 SQLite 查询体积、序列化体积和客户端 hydration 压力。

### Turbopack tracing warning

导出链路应收敛文件系统追踪范围，避免把整个项目纳入 NFT trace。重点检查：

- `app/api/projects/[id]/export/route.ts`
- `lib/export/*`
- 任何使用 `process.cwd()`、动态路径拼接或动态导入文件系统模块的代码

目标是让 tracing 定位到静态子目录，而不是项目根。

## 错误处理

- 局部数据请求失败时，仅在对应区块显示错误与重试，不影响整个工作区可用性。
- 轮询失败时保留当前画布显示，并降级为提示“状态更新失败，可手动重试”。
- 助手确认填充失败时不应触发其他区域的错误重置。

## 测试与验收

### 自动化测试

- 为新增的工作区数据拆分函数补充单元测试。
- 为轮询状态接口和局部刷新逻辑补充测试。
- 对画布、左树、助手相关源码断言测试做必要调整，确保新边界下源码与行为一致。

### 手动验收

- 打开项目页时，壳层先稳定出现，重组件随后可用。
- 修改方向/文案/图片配置时，不再出现整页闪动。
- 图片生成进行中时，页面保持稳定，仅对应节点状态更新。
- 助手确认填充需求卡后，仅相关卡片与树结构更新。
- 画布缩放、拖拽、聚焦节点时流畅度优于当前版本。

## 实施范围

本次实施包含：

- 页面分层与数据查询拆分
- 工作区局部失效与轮询改造
- 关键重量组件懒加载
- 画布局部渲染收敛
- 导出链路 tracing warning 修复

本次实施暂不包含：

- 全量替换所有 API 返回格式
- 引入新的第三方数据获取库
- 改动业务字段与交互流程

## 风险与控制

- 风险：工作区拆分后不同区域数据时序不一致。
  控制：使用明确的局部失效事件和最小共享状态，避免隐式同步。

- 风险：画布局部更新导致高亮或布局状态丢失。
  控制：将 UI 状态与服务端数据分离，合并时保留本地状态。

- 风险：拆分查询后出现字段遗漏。
  控制：为左树、画布、轮询分别写最小数据契约测试。
