## 测试方法

本次测试不是直接按各家官方原生 API 文档逐个验证，而是按当前项目的真实调用链路验证：

1. 直接请求洋葱网关的兼容接口：
   - `POST /v1/chat/completions`
   - `POST /v1/images/generations`
2. 再通过项目实际封装层验证：
   - `lib/ai/image-chat.ts`
   - `generateImageFromPrompt`
   - `generateImageFromReference`

因此，本结果回答的是：

> 这些模型在“当前项目 + 当前洋葱网关兼容层”下是否可用

而不是：

> 这些模型在各家官方原生 API 下理论上是否支持对应能力

## 逐模型结果

### 1. `gemini-3-pro-image-preview`

- 文生图：
  - `1:1` 成功
  - `3:2` 成功
  - `16:9` 成功
  - `9:16` 成功
- 图生图：
  - `1:1` 成功
  - `3:2` 成功
  - `16:9` 成功
  - `9:16` 成功

结论：

- 当前项目下唯一稳定通过“文生图 + 图生图 + 全比例”的模型

### 2. `gpt-image-1.5`

- 文生图：
  - `1:1` 成功
  - `3:2` 成功
  - `16:9` 失败
  - `9:16` 失败
- 图生图：
  - 当前实现中不支持

失败原因：

- 当前网关返回的约束为：
  - 支持的 size 只有 `1024x1024`、`1024x1536`、`1536x1024`、`auto`
- 当前项目为 `16:9` 和 `9:16` 生成的 size 分别是：
  - `1792x1024`
  - `1024x1792`
- 这两个值会被当前网关拒绝

结论：

- 可以保留在模型切换里，但应限制为仅 `1:1` 和 `3:2`

### 3. `qwen-image-2.0`

- 文生图：
  - `1:1` 成功
  - `3:2` 成功
  - `16:9` 成功
  - `9:16` 成功
- 图生图：
  - 当前实现中不支持

结论：

- 可作为仅文生图模型接入

### 4. `doubao-seedream-5-0-lite`

- 文生图：
  - `1:1` 成功
  - `3:2` 成功
  - `16:9` 成功
  - `9:16` 成功
- 图生图：
  - 当前实现中不支持

说明：

- 该模型在 `images/generations` 下可用
- 小尺寸会失败，必须满足最小像素要求
- 当前项目代码已为它单独映射更大的 size

结论：

- 可作为仅文生图模型接入

### 5. `gemini-3.1-flash-image-preview`

- 文生图：
  - `1:1` 失败
  - `3:2` 失败
  - `16:9` 失败
  - `9:16` 失败
- 图生图：
  - 全部失败

结论：

- 当前洋葱网关下不可用，不应放进前端可选模型列表

## 当前代码接入结论

本轮改动后，项目中的图片模型接入策略为：

- 已接入前端切换：
  - `gemini-3-pro-image-preview`
  - `gpt-image-1.5`
  - `qwen-image-2.0`
  - `doubao-seedream-5-0-lite`

- 未接入前端切换：
  - `gemini-3.1-flash-image-preview`

- 后端按模型能力自动路由：
  - `gemini-3-pro-image-preview` 走 `chat/completions`
  - `gpt-image-1.5` / `qwen-image-2.0` / `doubao-seedream-5-0-lite` 走 `images/generations`

- 图生图限制：
  - 仅 `gemini-3-pro-image-preview` 支持
  - 其余模型在图生图场景下会明确报错，而不是假成功
